# Design — implement-agent-core

## Context

onclaw's provider/secrets/store/onboarding layer is built and specced (`openspec/specs/{providers,onboarding}`). But `internal/agent/agent.go` is a 3-line stub and every provider adapter is a no-op stub — `onclaw run` prints "not implemented" and no inference occurs. This change implements the agent core on the eino ADK (`v0.9.9`, already a dependency), adds the first real provider adapter, builtin tools, a workspace model, a persona system, and a **named-agents** layer (multiple selectable agents, each with its own model and reasoning effort), integrating reference features A–D from `docs/references/agent-reference-takeaways.md`.

Constraints: single static binary (`CGO_ENABLED=0`), ~2 GB RAM target (`Concurrency=1`, `MaxContextTokens=8192` already set), pure-Go SQLite, single-user/single-process. Today only `github.com/cloudwego/eino v0.9.9` is a dependency — **eino-ext is absent** (see Risks).

## Goals / Non-Goals

**Goals:** a working streaming tool-using coding agent over a remote LLM; safe, workspace-scoped file/shell tools; a file-based persona; clean cancellation; a bounded context; offline-testable with a scripted fake model; **multiple named agents selectable per run, each choosing its model and reasoning effort**.

**Non-Goals:** memory/recall/reflect (M2); plugins/MCP/Lua (M3); full `sessions` table + `--resume`; Anthropic-native adapter; retry/failover wiring; prompt modes; **subagents / concurrent agents** (named selectable agents are in scope; concurrent or nested agents are not — see "Relation to takeaways §4" below).

## Decisions

1. **eino ADK `ChatModelAgent` (ReAct) over a hand-rolled loop.** Reuse the battle-tested loop + native middleware (summarization, cancel/interrupt, retry/failover). *Alternative: hand-roll the ReAct loop* — rejected (reimplements summarization + loop detection). Verified API: `adk.NewChatModelAgent(ctx, *ChatModelAgentConfig)`; `Run()` returns `*AsyncIterator[*AgentEvent]`; events carry a `MessageStream` (streaming) or `Message`, tagged by `Role` (`schema.Assistant`/`schema.Tool`).
2. **OpenAI-compatible adapter via eino-ext first; hand-rolled `net/http` fallback.** Covers OpenAI/Zhipu GLM/Ollama/gateways via `api_base`. eino-ext is absent and version-fragile (Risk #1); the fallback (~250–350 lines) preserves the no-CGO/static discipline and removes the dep risk. Decide path in Phase 0.
3. **Workspace as an explicit path, no `os.Chdir`.** Pass the absolute workspace to tools; file tools confine paths via `filepath.Clean` + prefix/`Rel` check; `shell` uses `cmd.Dir = workspace`. Matches GoClaw "resolve workspace". *Alternative: `os.Chdir`* — rejected (global process mutation).
4. **Persona via a small `context.go` loader** (optionally reusing eino `agentsmd` for `@import`). Files optional; fixed-order concat with a byte cap. *Alternative: DB-backed persona* — rejected for M1 (memory is M2).
5. **Minimal append-only transcript, no `sessions` table.** M1 writes per-turn events to `.jsonl` (`fsync` per turn); full sessions/resume is M2. Aligns with NullClaw "never hold the full session in memory".
6. **Summarization tuned to ~6k of the 8k budget** via the eino summarization middleware `Trigger`; a cheap ~4-chars/token counter (NullClaw pattern) if the default counter isn't adequate.
7. **Shell exec tiers + redaction-at-tool-boundary.** Shell policy `deny|allowlist|ask` (Feature A); a tool-execution wrapper/middleware sanitizes args and forbids secret rehydration (Feature D), generalizing the existing logging redaction (`internal/logging`).
8. **Named agents are a DB table, not files.** An `agents` table (name, provider ref, model + reasoning_effort overrides, system_prompt, tools, max_iterations, workspace) gives CRUD parity with `provider` profiles and atomic hot-reload via the existing fsnotify-on-DB watcher. `default_agent` is a `preferences` row, mirroring `default_provider`. *Alternative: agent files (`agents/*.md`)* — rejected for v1: CRUD/hot-reload parity with providers was valued over git-shareability. The **persona** stays file-based because it is a distinct concern ("who the user is", global) from agent behavior ("how this agent acts", per-role). This decision supersedes the earlier "no schema migration in M1" note.
9. **Agent selection resolves to an effective provider profile, leaving the adapter contract untouched.** Selecting an agent fetches its referenced provider profile, copies it, overlays the agent's `model` and `reasoning_effort` (into the copy's `settings`), and hands the copy to the existing `adapter.Build`. Credentials never move: an agent row holds no key; auth stays in the provider profile (`config_secrets`). This keeps the OpenAI-compatible adapter (Risk #1) and the stub-Anthropic constraint unchanged.
10. **Reasoning effort is a normalized enum, mapped per-provider at the adapter.** The agent row (and the `--reasoning` flag) carries `low|medium|high` (empty = provider default). The OpenAI-compatible adapter maps it to the provider's native field (`reasoning_effort`). Anthropic's `thinking.budget_tokens` is deferred with the Anthropic-native adapter. *Rationale:* one storage shape, provider-specific translation at the boundary.
11. **Agent-owned default workspace, resolved just below `--workspace`.** Each agent's default workspace is `~/.onclaw/workspace/<name>/` (created on `agent add`); resolution is `--workspace > agent.workspace > env > config > cwd`, so `--workspace` always wins and the agent default beats env/config/cwd. This is a deliberate, scoped override of `agent-reference-takeaways.md` §4 (see below).
12. **Global `USER.md` + per-agent persona/memory files.** One global `USER.md` at `~/.onclaw/USER.md` holds user-level facts shared by every agent (there is one user). Per-agent files — `IDENTITY`/`SOUL`/`CAPABILITIES`/`USER`/`AGENTS`/`MEMORY.md` — live in each agent's workspace and are NOT shared (`USER.md` exists in both scopes: global user facts + agent-specific user notes). Assembly is `[global USER.md] + [per-agent workspace files] + [agent.system_prompt] + [grounding]`. This is the narrow kind of "global" that does not reinstate a global persona: only user identity is shared; behavior/capabilities stay per-agent. The agent edits its workspace persona files with its normal file tools; the global `~/.onclaw/USER.md` is outside the workspace and stays hand-maintained.
13. **Single home `~/.onclaw/`.** All onclaw state — `config.yaml`, `onclaw.db`, `master.key`, `conversations/`, and `workspace/` — lives under `~/.onclaw/`. This supersedes the XDG split (`~/.config/onclaw` + `~/.local/share/onclaw`) for headless-Pi simplicity (no reliance on XDG env vars). Resolves Open Question #3 (consolidate: yes).
14. **A builtin default "master" agent bootstraps via an onboarding prompt + normal file tools.** The master is always present and is the initial `default_agent`, eliminating the cold-start "no agent available" failure. When its persona/memory files are empty, an **onboarding prompt** (a markdown file — default content in the `onboarding` delta) is prepended to its system prompt; the prompt says it has just woken up, encourages it to ask the user questions, and instructs it to record what it learns by editing its persona files with its **normal file tools** (`write_file`). No special tool, no separate interview mode. Once a persona/memory file is populated, the prompt is no longer injected. *Rejected: a privileged `save_persona` tool / hardcoded interview mode* — unnecessary; a prompt file plus the existing file tools suffice. *Consequence:* file tools are workspace-scoped, so the agent edits per-agent workspace files, not the global `~/.onclaw/USER.md`, which stays hand-maintained.

15. **Per-file persona templates + flag-based onboarding trigger.** Each persona/memory file (`IDENTITY`/`SOUL`/`CAPABILITIES`/`USER`/`MEMORY`/`AGENTS.md`) ships a default template under `internal/agent/templates/`, embedded; `onclaw init`/`agent add` seed each file from its template (non-destructive) instead of empty. Because files now ship with content, onboarding can no longer fire on file emptiness — so the system tracks an `onboarding_completed` preference and injects the onboarding prompt only while it is unset, marking it complete when the onboarding interaction concludes (supersedes the emptiness-based trigger clause in Decision 14). Templates are the single source of default content (no hardcoded constants), unifying with the onboarding-prompt-as-file decision.

16. **Builtin tools live in an extensible `internal/agent/tools/` sub-package behind a `Tool` interface.** Each builtin tool (`read_file`/`write_file`/`list_dir`/`shell`) is implemented in its own file as a `Tool` whose `Build(scope)` returns a workspace-scoped eino `tool.InvokableTool`; a `registry.go` holds the single list of builtins and a `Builtin(scope)` factory assembles them. This satisfies the project's one-concern-per-file rule (`.claude/rules/coding-style.md`) and makes adding a tool a purely additive change (implement the interface + register — no edits to existing tool files). Secret masking stays a cross-cutting concern: a redaction decorator wraps every tool inside `Builtin`, so a newly registered tool inherits masking automatically; the exported `tools.Redact` is reused by the transcript so the secret patterns live in exactly one place. *Alternative: keep all tools in one `tools.go` in the `agent` package* — rejected (violates one-concern-per-file; not extensible). *Alternative: self-registering tools via package `init()`* — deferred (global `init()` side effects are undesirable in a low-resource single-process binary; an explicit `registry.go` slice is simpler and grep-friendly). This supersedes the implicit single-file tool layout assumed by the original §4 tasks.

## Relation to agent-reference-takeaways.md §4

The takeaways explicitly rejected multi-agent `AgentRegistry`, per-agent session DBs, and per-agent tool registries as 2 GB-budget killers. The named-agents layer here is compatible with that verdict: there is **no registry of concurrently running agents** — `Concurrency=1`, only one agent runs at a turn, sessions stay in one shared `conversations/` dir (the M2 `sessions` table gains only an `agent` filter column), and tools are built from one shared registry scoped by the resolved workspace. What is borrowed from GoClaw is the *definition* shape (provider ref + model + workspace per agent), not its multi-tenant runtime. This scoping is the reason the override is safe.

## Risks / Trade-offs

- **[Risk] eino-ext version vs eino v0.9.9** (cached `components/model/openai@v0.1.13` pins v0.7.13) → *Mitigation:* Phase 0 `go get eino-ext/libs/acl/openai@latest`; if MVS conflicts, hand-roll. No silent downgrade of eino core.
- **[Risk] eino API drift** (`Next()`/`Recv()` names, the `ToolsNodeConfig` tools field, summarization `Trigger` shape) → *Mitigation:* confirm exact signatures from module-cache source at implementation start; a stub/fake model keeps tests offline.
- **[Risk] 2 GB budget with eino overhead** → *Mitigation:* keep `Concurrency=1`, stream (no full buffering), bound context to 8k, one agent at a time.
- **[Risk] reasoning-effort field name differs per OpenAI-compatible endpoint** (OpenAI `reasoning_effort` vs Zhipu/GLM variants) → *Mitigation:* confirm the target field in Phase 0; map the normalized enum at the adapter, fail closed (no effort sent) if unrecognized.
- **[Trade-off]** hand-rolled adapter = owning the tool-calling schema mapping if eino-ext fails — more code, full control, zero dep risk.
- **[Trade-off]** DB-stored agents are not git-shareable; project-local team-shared agents (a `.onclaw/agents/` layer) are deferred.
- **[Trade-off]** persona files are hand-edited only (no auto-learning) until M2 memory.

## Migration Plan

Additive: stub → real. **Schema migration is now in scope:** `CREATE TABLE agents (...)` and a `default_agent` `preferences` row. M2 later adds an `agent` column to `sessions`. The migration is additive (new table + preference key); existing providers/secrets/onboarding rows are unaffected. `onclaw init` creates the personal persona files (`IDENTITY`/`SOUL`/`CAPABILITIES`/`USER.md`) **empty** and seeds `AGENTS.md` with default content, only when absent (non-destructive, consistent with the `onboarding` spec). `onclaw agent add <name>` creates the agent row and its `~/.onclaw/workspace/<name>/` directory. Rollback = revert the change + drop the `agents` table.
